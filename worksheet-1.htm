4.     Open up the conference.css file (as shown above).

5.     Notice how the CSS file provides explicit widths for many of the styles. For those unfamiliar with CSS,[2] each style is either a style class or style identifier. They are selected (referred to) within the HTML file (in this case index.html) using class and id attributes. Your task is to change from explicit to relative widths using percentages (next steps).

6.     The body CSS style defines the overall width of the screen. Its currently using fixed widths for sizing elements and margins. We are going to use percentages to make the design fluid when we resize the browser. In the following design we have replaced absolute pixel width values with percentages.

The overall screen-design width is 920px + 15px + 15px = 950px (body elements width + its left margin width + its right margin width). Assuming that the overall screen width is 100%, the body width as a percentage is calculated with: 920 / 950 * 100 = 96.842105%. The left and right margin percentages will be: 15 / 950 * 100 = 1.5789474%, that is we can set the bodys margin rule to use the value 1.5789474%. As well as scaling the left and right margins, this will also scale the top and bottom margins to the same value as the left and right margins, keeping the margin values the same on all sides.

Your first task is to copy the percentages from the following table into the CSS file, replacing absolute pixel values by percentages for width, padding-left, padding-right, margin-left, margin-right and margin. Only deal with the CSS elements referenced in the design. For this exercise, I have rounded down the percentages to two decimal places to make typing the values easier.

Make sure you save the changes you make. You can do this with ctrl-s (Windows) or cmd-s (Macs) or via the File menu or Save button at the top of the Bluefish window.

7.     Take a look at the .logo img style in the conference.css file. This rule applies to the <img> element in index.html (which is a child element of the logo div element). Replace its width style rule with max-width: 100%, i.e. use 100% of the width of the <img> element, but no wider. Note that the image can now shrink less than 250px (its implicit actual width).

If you want to specify a lower bound then use a min-width rule, e.g. min-width: 200px. This stops the image becoming too narrow.  However, when you shrink the browser window notice how the <img> element overflows out its logo parent container and into the header area. Also, you could change the height to be auto to maintain the aspect ratio (but lets not do this for our design).

8.     Replace the font-family rule in the body style with:

font: 100% "Helvetica Neue", Helvetica, Tahoma, Arial, sans-serif;

This ensures that the font size will be 100% of that set for the browser.[3] The user may adjust this within the browsers settings, e.g. for those preferring a larger or smaller font, and also by zooming in or out.

9.     We should also change the font sizes of headings and the header.page_header height so that they are relative to the default font size; e.g. if the user changes the default font size then we want the page header to grow and shrink.

a.     Add the following h* styles just below the body style:

h1 {

    font-size: 2.5em;

}

h2 {

    font-size: 1.5em;

}

h4 {

    font-size: 1.08em;

}

b.     Change the p and p, li styles to be:

p {

    margin: 0 0 1.3em 0;

}

 

p, li {

    line-height: 1.3em;

}

c.     Change the header.page_header style so that its height rule is now of size 12.5em rather than the absolute pixel size.[4]

d.     Change the .logo img style so that its height is now 12.5em.

e.     Change the header.page_header h1 style so that its height is 3.3em and its padding-top is 1.2em.

f.      Finally, change the font size in the aside style to have the value 1.25em.

Step One: Using media queries

In this step we will look at a new version of the Conference web site, one that uses media queries to add alternative structural CSS for different browser and mobile screen dimensions.

6.     Using RubyMine (or similar), navigate to the with-media-queries-solution folder and open up the conference.css file. Lets examine the file.

7.     We want to leave styles that apply to all screens at the top of the CSS file, and only extract and insert the differences that are screen-specific to their appropriate media query style sheet. Have a quick scan through this top part of the file. Much of it will be familiar from the previous worksheet.

8.     Eventually, if you scroll a little further down you will see:

/* Normal width desktop browsers, some tablets */

/* LARGE SCREEN MEDIA QUERY */

/* @media screen and (min-width: 685px) { */

@media screen and (min-width: 42.8125em) {

                  styles removed

}

 

This is where we define styles that apply to screens that are at least 685 pixels wide. We have changed the unit of measurement to ems[6], based on a default font size of 16px and where 1em equals 16px (685 / 16 = 42.8125em). We have done this so that if the user zooms in or out within the browser the media query breakpoint width will scale automatically, e.g. zooming in by 100% would mean that 1em was now 32px and the media query min-width breakpoint for large screens would now be 1370px (685px * 2), i.e. the browser window needs to be double the width before the media query breakpoint kicks in.  Why is this useful? It means that the larger text in titles etc is given more width.[7]

 

The media type is screen. We can also define media queries that apply to printing, projection, TV etc

 

A little further down we have the media query for medium sized screens/browsers:

 

/* Narrower browsers, some tablets */

/* MEDIUM SCREEN MEDIA QUERY */

/* @media screen and (min-width: 527px) and (max-width: 684px) { */

@media screen and (min-width: 32.9375em) and (max-width: 42.75em) {

                 

}

 

These styles only apply for browsers/screens that are between 527 pixels and 684 pixels in width.

If you scroll further still you will find the media query for narrow screens/browsers:

 

/* Narrow browsers and smartphones */

/* @media screen and (max-width: 526px) { */

@media screen and (max-width: 32.875em) {

                 

}

Step Two: Running from a web server

You will be able to undertake this step if you have installed a web server or are connected to IS central filestore. This will be the case on our B57 Macs. Copy the code folder to our central server public_html folder. You can access the content of that folder via your browser with http://users.aber.ac.uk/your-uid.

Providing a web site from a web server gives you the opportunity of running and testing the site on a mobile device. It also makes accessing the site from the emulator a more realistic experience.

1.     Starting/configuring the web server (only if youre not using our central server web server, but rather your own server): Here are some configuration steps that you may have to perform, depending on which web server you are running:

a.     Mac users: Most Macs have an Apache web server built in. If you are using one of our Macs then it will be enabled and running. For your own Mac see our environment setup instructions on how to enable it (it will be disabled by default). If using the Macs built-in Apache server, then copy your code folder to your Sites folder. You can get to this by using the Finder Go menu and selecting the Home item. Alternatively, you may have installed MAMP on the Mac (see below).

b.     Windows/Linux users who have XAMPP installed: Start XAMPP. Then click on the Apache Start button. Copy the code folder to XAMPP Apaches document root folder: [XAMPP-installation-folder]/htdocs.  See our setup document for more details.

c.     MAMP users: If using MAMP, when started, click on the Preferences button. Start by switching off the cache. This can be done by selecting the PHP tab and setting the cache to ---. Now select the Apache tab and then navigate to your code folder. Click OK and wait for MAMP to restart Apache. 

2.     Start a browser or create a new browser window and point it to either http://localhost/~user-name (if using a Macs built-in Apache server) or http://localhost (if using XAMPP) or http://localhost:8888/ (if using MAMP). This should display the code folders directory structure.

3.     Click on the with-media-queries-solution link. This should display the same screen you saw earlier, but this time provided via the web server.

Step Three: Running on the mobile emulator

Using an emulator is a quick way of checking the appearance of a web site without having to use a real device and circumnavigates issues related to firewalls. The Opera Emulator emulates the Opera Mobile browser for a range of devices.

1.     If you have one, try accessing the web server from your mobile device (its MAC address will have to be registered with IS to be able to access your computer via the IS wireless network otherwise youll need to use your mobile operators network). Access the website using http://users.aber.ac.uk/your-uid/code/with-media-queries-solution.

2.     Check to see if the mobile version of the browser picks up and displays the web site correctly. It should display the mobile version of the website.

3.     Open up the index.html file and notice the following in the <head> element:

<meta name="viewport" content="width=device-width, initial-scale=1"/>

This will ensure that the layout viewport width is the same as the screen width. The initial-scale attribute ensures that no zooming will be applied automatically; this does not prevent the user from undertaking zooming using the touch screen etc.

This will ensure that the mobile version of the website should appear:

Figure 4. Design when run on a Samsung Galaxy SII Opera Mobile browser.

Try commenting out the viewport command and notice how your mobile browser now displays the desktop version of the website but zoomed out.

Step Four: Examining and running the mobile-first version

12.  Point your browser to the folder with-media-queries-mobile-first-solution.

13.  You should not notice any difference between this version of this site and that developed for Step One; it is simply using separate CSS files and media queries embedded in the HTML file.

14.  Use RubyMine to navigate to with-media-queries-mobile-first-solution folder. Open up its index.html file (Figure 5). Notice how the conference-mobile-first.css file is included without using a media query, whereas the link commands for the other two CSS files incorporate appropriate media queries; there are no media query statements incorporated within the CSS files.

Those browsers that do not support CSS3 media queries will ignore these media queries, applying only the mobile-first CSS. Notice that for Internet Explorer 8 and earlier and where not also running on a mobile device, then the IE conditional statement brings in the large screen CSS. Clearly, this could be configured to work more appropriately with IE.
